## THE DECIDED SCOPE (do not relitigate)

**1. Persistent-storage request.** On app init (once), call `navigator.storage.persist()` (guard
for availability). Surface the resulting status in Settings as a small read-only line (e.g.
"Storage: PERSISTENT" / "Storage: BEST-EFFORT — export backups regularly"). No nagging, no modal.

**2. "Export full backup" button in Settings.** Exports ALL Dexie tables as one JSON via the
share-or-download mechanism W22 built. Requirements:
- Enumerate tables DYNAMICALLY (`db.tables`) — so when W23 adds notes stores, they are included
  automatically with zero changes to this feature. Do not hardcode table names.
- Shape: `{ format: 'combatos-full-backup', version: 1, exportedAt, schemaVersion: db.verno,
  tables: { <tableName>: [rows...] } }`. This file is both disaster recovery and the future
  Supabase migration seed — shape stability matters; pin it with a test.
- Filename: `combatos-backup-YYYY-MM-DD.json` (plain local date stamp).
- Reuse/generalize `app/src/utils/checklistShare.js` (e.g. extract a `shareOrDownloadJson(data,
  filename)`); the existing checklist export must keep working unchanged.
- RESTORE IS EXPLICITLY OUT OF SCOPE (merge semantics are the Supabase era's problem — document
  this in a code comment).

**3. "Last backup" hint.** After a backup that was actually delivered (NOT cancelled — the share
flow's user-cancel/AbortError must not count), store a timestamp in the existing key-value
`settings` store. Settings shows "Last full backup: N days ago" / "never". If the share util can't
currently distinguish delivered-vs-cancelled, refactor it minimally so it can.
